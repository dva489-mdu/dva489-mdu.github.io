DVA489 - Web security

The course is split into two parts. The first part consists of practical application development, attacks and analysis. The second part consists of an excursion into future protection mechanisms.

The assignments of the course will be made available in good time before the introductory lecture corresponding to each assignment.

This course covers a lot of material. Some of it will be introduced in the introductory lectures, but you are required to do a lot of reading on your own. To your aid there will be links to relevant resources and you will have the opportunity to ask questions during our supervision sessions. If your questions cover a larger topic, please contact us in advance to allow us to prepare additional material.

Part 1: Security Practice

This part consists of five applications with associated attacks. Each of the five parts should be equivalent to roughly 1 credit, corresponding to about 3 days of work. In order to gain full understanding of the attacks its important to understand the workings of the applications. Thus, you are going to develop the applications to attack yourselves.

Example solutions

To help you on the way there are relatively detailed assignment guides, with online example solutions. You are free to take inspiration from the example solutions in terms of the content they serve, including HTML, CSS, JavaScript, etc. Indeed, the guides expect you to examine and reverse engineer their solutions. Feel free to take inspiration, or reuse and improve!


Example solutions

You are allowed to use any information you can get from the example solutions, including the code if you manage to successfully exploit them. I've made an effort to secure them, but there are no guarantees!

Early protection

Since you will be attacking your own web applications there is a risk that the code you write is not vulnerable to attack. The guides try to alleviate this problem by guiding you towards a believable solution containing common security flaws. The intention is that, if you follow the instructions closely without adding additional security measures of your own, you should end up with an application open to selected attacks.

Early protection

Abstain from prematurely hardening your application! This removes the fun of successfully attacking the application!

Use of 3rd party libraries

To ensure a detailed understanding of the inner workings of the vulnerabilities, and the protection mechanisms it is important you implement things yourselves. Thus, any use of NPMs that are not explicitly allowed is forbidden and will result in the rejection of the assignment.

3rd party libraries

Inclusion of libraries that are not part of the core nodejs runtime will result rejection of the assignment!

Submission requirements

This course uses GitHub Classroom for all assignments. Thus, you hand in by creating a pull request from your working branch into you main branch. Instructions on how to do this are available in the GIT repository of each assignment.

The GIT repository should contain you attack reports in addition to the source code of your application as a proper NPM package. This means that there should be a package.json file that describes the package and (for the assignments that makes use of 3rd party NPMs) lists all included NPMs. Do not pollute the GIT repo with the node_modules directory. See An introduction to the NPM pakcage manager for more information.

Furthermore, the changes made to protect agains each attack should be commited in its own commit with a clear commit message. Repositories that only contain one big commit will be rejected.

Malformed submissions

Failure to adhere to the above will lead to rejection of the assignment.
